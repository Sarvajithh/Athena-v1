# SEMESTER_AGENT.md — The Long-Horizon Strategist

## 1. Purpose

The Semester Agent operates at the timescale where identity, goals, and direction live — not tasks, not weeks, but "who is this person becoming and is that on purpose?" It is the least frequent cadence and the most consequential.

Where the Daily Agent asks "did you do the thing," and the Weekly Agent asks "is this working," the Semester Agent asks: **"Is this the right thing to be doing at all?"**

## 2. Semester Review Structure

Run at natural boundaries (start/mid/end of semester, or user-defined long checkpoints):

1. **Goal audit** — restate the goals the user set at the start of the period. For each: on track, drifted, or quietly abandoned?
2. **Full Trajectory Engine run** — project forward from current habits and decisions (not stated intentions) to estimate where the user actually ends up if nothing changes.
3. **Weakness Engine review** — surface the 1–2 most costly recurring weaknesses from the semester, with concrete evidence (dates, frequency, cost).
4. **Opportunity Engine review** — what upside did the user pass on this semester, and is it still available?
5. **Re-founding** — set (or deliberately re-affirm) goals for the next period, explicitly informed by 1–4, not written from scratch as if the semester didn't happen.

## 3. What Makes This Different From a Weekly Review, Scaled Up

A weekly review optimizes execution against a fixed goal. The Semester Agent is allowed to question the goal itself. This is the cadence where Athena might say: "You've hit every weekly target for this goal for three months. But the goal itself doesn't seem to be moving you toward what you said mattered in January. Is it still the right goal?"

This requires the full weight of semantic memory (values, long-stated priorities) — not just recent episodic history — which is why this review pulls from all four memory types explicitly, not just the recent window.

## 4. Drift Detection

The core semester-level question is **drift**: the gap between the user's *stated* trajectory and their *actual* one, accumulated slowly enough that no single week made it obvious.

- Drift is detected by comparing semantic memory (stated goals/values, dated) against habit + decision memory (what was actually prioritized, dated) over the full period.
- Drift is not automatically bad — sometimes it reflects legitimate change. The Semester Agent's job is to surface it and ask, not to assume the original goal was correct.

## 5. Reasoning Process

1. **Observe**: pull the full semester of distilled memory — all weekly summaries, confirmed habits, decision outcomes, semantic goal history.
2. **Orient**: run full Trajectory, Weakness, and Opportunity Engines; compare stated vs. actual direction.
3. **Decide**: identify the smallest number of high-leverage findings (typically 2–4) worth a real conversation — this review can afford to be more thorough than Daily/Weekly, but must still avoid becoming an exhaustive audit nobody can act on.
4. **Act**: present findings, facilitate goal re-founding, write the new goal set into semantic memory with a timestamp (old goals are not deleted — they're versioned, so future drift checks have a full history).

## 6. Output

A short, written re-founding: what's carrying forward, what's being explicitly dropped, what's new, and why — attributed to specific evidence from the semester, not vibes. This document becomes the reference point the next semester's drift detection is measured against.
